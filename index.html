配置DispatcherServlet

DispatcherServlet是spring mvc的核心，它负责将请求路由到其他组件中。
传统的DispatcherServlet配置在web.xml中，现在更推荐的配置是java配置。
使用java配置需要servlet3.0的支持，需要tomcat7及以上版本。

public String processRegistration(@Valid Spitter spitter, Errors errors) {

        return "registerForm";
    }
    spitterRepository.save(spitter);
    return "redirect:/spitter/" + spitter.getUsername();

责任链模式

发表于 2016-07-27

作用

  用于多个对象对请求的处理，每个对象都有处理的机会，需要根据具体情况来决定由谁处理请求。

实例

场景

  在一个管理类的游戏中，存在着几种请求，征税、审问犯人、防御城池，这三种都是由玩家发出请求，然后由具体的执行者执行，
分别为军人征税、警长审问犯人、指挥官防御城池。

分析

  不同的请求需要由不同的执行者执行，可以先判断请求类型，然后直接由对应的执行者来执行，这当然可以，但是这样请求和执行者属于紧耦合，
更好的方法是采用责任链模式，将所有的执行者串成一串，由头开始依次往后传递，知道有执行者执行请求。

设计

  最关键的是把所有执行者串成链，然后依次传递请求，这非常适合装饰者模式来完成。

实现

首先是请求的定义

然后是责任链的装饰者模式定义

指挥官、警长、军队继承执行者基类

请求的发出者是国王

国王发出请求，请求会沿着责任链传递，知道请求被处理或者没人处理

由合适的对象处理请求

总结

  • 责任链模式就是所有可能处理请求的对象形成一条链条，当请求来临时，沿着这条链条依次传递知道请求被处理或者没有对象处理。
  • 将请求和处理对象耦合度大大降低。
  • 对于处理对象链来说，拓展非常方便，增加或减少处理对象几乎不用修改代码。
  • 责任链的传递规则是透明的，可以按照具体需求制定规则。
  • 如果处理对象数量过多，可能会影响性能。

建造者模式

发表于 2016-07-17

作用

  用于分步骤的创建一个对象的各个部分，最终创建出一个复杂对象。

实例

场景

  现在需要创建一个英雄，英雄有自己的名字、职业、武器、盔甲、还有一些外观的差异。

分析

  像名字、职业是一个英雄必须具有的主属性，而武器、盔甲等属于英雄的附加品，应该属于可有可无的，而且可以随时增加、修改、卸下。

设计

  应该提供一个建造类来专门负责英雄的创建工作。其中名字和职业是必须提供的，其他部件可以提供相应的方法随后任意时刻创建。

实现

首先是英雄的几个部件类，包括职业、武器、盔甲、发型、发色，这些都是属于枚举类型

然后是英雄类和英雄建造类

创建的时候，利用创建类来完成部件的创建

输出的英雄的各个部件

总结

  • 建造者模式将一个复杂对象分离成一个个部件分步骤创建。
  • 建造者模式可以限制部件的生成顺序，也可以利用已经生成的部件来对后续部件做出约束。
  • 客户使用简单，只需要制定部件的类型和顺序即可生成复杂对象，方便扩展和修改。

桥接模式

发表于 2016-07-10

作用

  用于将接口和实现脱耦，使得两者可以独立的变化。

实例

场景

  现在要做一个游戏的法宝系统，法宝有很多种类，每种有多种。需要客户能方便的改变法宝，系统可以很方便的修改和增加法宝。

分析

  先给法宝做一个抽象，然后每一类法宝作一个抽象，具体的法宝实现此类法宝的抽象类。为了接口和实现脱耦，需要将接口和实现分离，形成两个继承体系。
只在运行时将接口和实现绑定。

设计

  首先是接口实现分离，类似于代理模式。接口是调用实现的方法，两者在运行期绑定。接口和实现都有一个公共抽象类，这两个类的方法应该一样。具体的法宝是扩充这个抽象类实现的。
接口和实现只要保证是属于一类的抽象，就可以实现独立的变化，随意的匹配组合。uml图看上去就是两个继承体系中间的1：1关系，就像一座桥，这也是桥接模式的由来。

  • 使抽象部分和实现部分分离，可以独立的变化。

  • 具有非常高的复用程度和灵活性，但有一定的复杂度。

适配器模式

发表于 2016-07-06

作用

  用于接口的适配，将一个类以不同的接口进行封装，以达到和客户匹配的目的。

实例

场景

  海盗船长开着海盗船四处抢劫，有敢于反抗的就会开火轰趴他们。船长当然会航行和开火两项技能，这两项功能战船都能提供，但是现在
海盗遇到海军战败了，战船被烧，四处逃亡，最终发现了一艘渔船。虽说和威风的战船没法比，那也必须会开啊。

分析

  船长会开船和开炮，渔船只能航行却不能开炮，那简单啊，给船长一个开火按键，告诉他这就是开火就可以了。这样船长还和以前一样开着战船，没有任何
变化。

设计

  这就很简单了，把渔船和战船的接口改成一样就好了。但是渔船已经成型，直接改动会不稳定，所以我们只需加一层封装来实现接口的适配就可以了。
这就是适配器模式。

实现

然后是船长会操作战船移动和开火

关键的时候来啦，把渔船适配成战船

船长继续和以前一样，一遍航行一遍开炮

总结

  • 适配器模式就是将一个类或者模块封装起来，这个封装层提供给外接不同的接口已适应客户
  • 一个类可以提供不同的适配层提供给不同喜好的客户，提高了代码的复用度
  • JDBC就是非常好的适配器模式的使用案例，不同的数据库虽然实现不一样，但是适配层接口都保持
    一致，客户改变数据库时不需要更改代码，只需要替换具体驱动即可
  • 实际的使用中，比如维护一个很乱又没有文档的项目时，可以将很的模块隔离开来，提供一个适配层给外界使用，能在不出错的前提下增加清晰度

工厂方法

发表于 2016-07-02

作用

  用于创建不同产品，对创建过程进行了抽象，使得客户代码可以很方便的更改产品或是新增产品。

实例

场景

  现在需要为不同种族制作武器，精灵族和兽族的武器肯定是大不相同，需要一个设计让客户代码可以方便的选择具体创建什么种族什么类型的武器。

分析

  需要创建武器，需要确定什么种族和什么类型的武器。这两部可以进行抽象，抽象成锻造工人和武器，锻造工人分为精灵族和兽族，根据武器类型可以锻造对应种族
的武器。

抽象工厂

发表于 2016-06-29

作用

  用于创建相似结构、不同系列的产品。抽象工程模式将设计实现分离，大大提高了复用率。

举例

需求

  游戏开发需要创建不同的王国，王国需要的元素有国王、军队、城堡。不同种族有不同风格的王国，
比如有精灵族、兽族、不死族。现在需要根据种族来创建对应风格的王国。

分析

  不同的种族的构成相似，都是由国王、军队、城堡构成。一个种族内部的风格保持一致。
把王国作为产品，种族就是相应系列。这里抽象出产品、系列、部件，产品由系列唯一确定，系列
由一套相同风格的部件确定。每个部件为一个抽象，不同风格为不同实现。这样，整体的设计就慢慢
清晰了。

  • 抽象工厂模式提供一个创建一系列产品或一种风格的接口，与具体类分离，客户不需要知道具体的创建过程。
  • 抽象工厂模式主要适用于创建产品族，具有通用的抽象层次，代码复用率高。是所有形式的工厂模式中最具一般性的。
  • 抽象工厂可以很方便的增加产品族，但是改变产品族的构成接口比较麻烦。所以适用于结构稳定的产品族的创建。
